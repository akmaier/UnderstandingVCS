## Goal
The jutari counterpart of the (jaxtari) E0-2 recorder: given a game, deterministically
replay N frames and record the per-frame **state trajectory** — the 128-byte RAM tape (and
optionally TIA registers / screen) stacked into a `(T, ·)` array — for the Phase-A tuning
curves / dim-reduction (E3) and the Phase-C SAE activation capture (E5). Built on the
verified `common/jutari_oracle.jl` (load/boot/replay/snapshot), so it inherits the Paper-1
bit-exact path. This is shared infrastructure; downstream pilots import it.

## Definition of Done
- [ ] `jutari_record.jl` exposes `record_trajectory(game; frames, fields)` returning a
      `(T, n)` Float64/UInt8 matrix (RAM tape by default) + a frame index, reusing
      `jutari_oracle.jl`
- [ ] writes a trajectory artifact to `common/out/traj_<game>.{npz}` (numpy-loadable via the
      existing dependency-free writer) + a tiny JSON sidecar (game, frames, fields, shape)
- [ ] deterministic: two fresh recordings byte-identical (asserted in `test_jutari_record.jl`)
- [ ] runs on jutari from a documented command (`julia --project=<primary>/jutari ...`)
- [ ] nothing outside `file_scope`; emulator core untouched; no shared-venv pip
- [ ] committed + pushed to main (rebase-before-push)
- [ ] `status: done`

## Notes / handoff
The jutari substrate is the verified fast path (jaxtari eager is ~205× slower — SCRUM §7).
Keep it minimal: a thin trajectory wrapper over `jutari_oracle.jl`'s replay+snapshot. E3-0
and E5-0 depend on this. The jaxtari E0-2 recorder is retained for the cluster SOFT-STE
batch path.

### Done (P2 S3)
- `tools/xai_study/common/jutari_record.jl` — `record_trajectory(game; frames, fields,
  actions)` deterministically replays via `JutariOracle.load_pong_env` + `env_step!` and
  stacks the per-frame state into a `(T, n)` `UInt8` matrix. Fields: `ram` (128-byte tape,
  default), `tia` (64-byte register file), `screen` (210×160 framebuffer, row-major
  flattened → 33600). `write_trajectory` emits `common/out/traj_<game>.npz` (keys `tape`,
  `frame`, `widths`) via the oracle's dependency-free NumPy writer + a `traj_<game>.json`
  sidecar (game, frames, fields, widths, shape, commit, …) written by a tiny inline JSON
  serializer (no package added to the shared jutari env).
- Run: `julia --project=/Users/maier/Documents/code/UnderstandingVCS/jutari
  tools/xai_study/common/jutari_record.jl --game pong --frames 60 --fields ram`
  → tape shape `(60, 128)`; artifact `tools/xai_study/common/out/traj_pong.{npz,json}`.
- Test: `julia --project=.../jutari tools/xai_study/common/test_jutari_record.jl`
  → **18/18 pass**: shape `(60,128)`, two fresh recordings byte-identical (determinism),
  ram+tia stacking `(60,192)` with widths `[128,64]` and matching RAM slice, npz/JSON
  written + valid ZIP. Verified numpy round-trip (`np.load` → `tape (60,128) uint8`,
  `frame 1..60`, `widths [128]`) and screen row-major round-trip.
